<h1>📘 BOAS PRÁTICAS – SWEET AFFINITY BACKEND</h1><h2>🎯 Objetivo</h2><p>Garantir padronização, escalabilidade e previsibilidade no backend, evitando divergências de arquitetura, imports e regras de negócio.</p><h1>🧱 1. PADRÃO DE ARQUITETURA</h1><h2>✔ Stack oficial</h2><ul><li>Fastify (API HTTP + WebSocket)</li><li>TypeScript (NodeNext)</li><li>Drizzle ORM (único ORM permitido)</li><li>PostgreSQL (banco principal)</li><li>Redis (cache / presença / rate limit)</li><li>Socket.io (tempo real)</li><li>MinIO (storage compatível S3)</li></ul><h2>❌ NÃO UTILIZAR</h2><ul><li>SQL raw com pgPool</li><li>ORMs alternativos (Prisma, Sequelize etc.)</li><li>Dependências diretas entre controllers</li><li>Lógica de negócio dentro de socket ou routes (quando possível)</li></ul><h1>📁 2. PADRÃO DE IMPORTS (CRÍTICO)</h1><h2>✔ SEMPRE usar .js nos imports internos</h2><p>import { db } from "../db/index.js";<br /></p><h2>❌ NUNCA usar .ts</h2><p>import { db } from "../db/index.ts"; // ERRADO<br />⚙️ 3. BANCO DE DADOS</p><h2>✔ padrão oficial</h2><ul><li>Drizzle ORM obrigatório</li><li>migrations via drizzle-kit</li><li>schema único em /src/db/schema.ts</li></ul><h2>❌ proibido</h2><ul><li>pgPool direto em controllers</li><li>queries SQL soltas</li><li>schema duplicado</li></ul><h1>🔐 4. AUTENTICAÇÃO E SEGURANÇA</h1><h2>✔ padrão</h2><ul><li>JWT obrigatório em rotas protegidas</li><li>request.jwtVerify() sempre que necessário</li><li>role system baseado em:</li></ul><p>profileType: "user" | "admin"<br />❌ proibido</p><ul><li>confiar em userId vindo do frontend sem validação</li><li>autenticação no socket sem verificação de token</li><li>lógica de permissão duplicada</li></ul><h1>🧠 5. REGRAS DE NEGÓCIO</h1><h2>✔ separação obrigatória</h2><ul><li>profileType → segurança do sistema</li><li>relationshipType → regra de negócio (matching)</li></ul><h2>❌ proibido</h2><ul><li>usar “Baby / Daddy / Mommy” como role de sistema</li><li>misturar segurança com matchmaking</li></ul><h1>📦 6. ENVIRONMENT (.env)</h1><h2>✔ padrão obrigatório</h2><ul><li>.env.example → sem segredos</li><li>.env.local → dev real</li><li>.env → produção</li></ul><h2>❌ proibido</h2><ul><li>credenciais reais no .env.example</li><li>hardcode de secrets no código</li><li>compartilhamento de JWT_SECRET público</li></ul><h1>📡 7. WEBSOCKET (SOCKET.IO)</h1><h2>✔ padrão</h2><ul><li>autenticação obrigatória no handshake</li><li>validação de sender_id no backend</li><li>mensagens sempre persistidas antes de emit</li></ul><h2>❌ proibido</h2><ul><li>confiar em dados enviados pelo client sem validação</li><li>lógica de banco dentro do socket sem controle</li></ul><h1>🧩 8. CONTROLLERS</h1><h2>✔ padrão</h2><ul><li>controllers devem conter apenas:<ul><li>validação</li><li>chamada de service</li><li>resposta HTTP</li></ul></li></ul><h2>❌ proibido</h2><ul><li>lógica complexa dentro de controller</li><li>SQL direto</li><li>regras de negócio duplicadas</li></ul><h1>🚀 9. PADRÃO DE CÓDIGO</h1><h2>✔ obrigatório</h2><ul><li>TypeScript estrito</li><li>async/await padrão</li><li>try/catch em todas rotas críticas</li><li>logs via fastify.log</li></ul><h1>📁 10. NOMENCLATURA</h1><h2>✔ padrão oficial</h2><ul><li>users → autenticação</li><li>profiles → dados públicos</li><li>photos → mídia</li><li>messages → chat</li><li>admin → controle interno</li></ul><h1>⚠️ 11. REGRA DE OURO DO PROJETO</h1><p>❗ “Nenhuma regra de negócio deve depender de SQL manual ou código duplicado em múltiplos arquivos.”</p><h1>🧠 12. PRINCÍPIO ARQUITETURAL</h1><h2>✔ sempre seguir:</h2><ul><li>Separação de responsabilidade</li><li>Fonte única de verdade (Drizzle schema)</li><li>Validação no backend (nunca confiar no frontend)</li><li>Evolução incremental (sem refatorações caóticas)</li></ul><h1>🔥 RESUMO FINAL</h1><p>Este projeto segue o padrão:</p><p><strong>Fastify + Drizzle + TypeScript ESM + arquitetura modular + tempo real via Socket.io</strong></p>
